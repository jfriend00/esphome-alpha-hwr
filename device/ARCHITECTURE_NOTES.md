# Architecture Notes
Notes about things I might discuss with the upstream maintainer

## Pump Speed
The current upstream architecture relies on storing the intended pump speed in the pump (probably modelled after the phone app).  I prefer to have Home Assistant be the master for the intended pump speed and then use that speed when starting the pump.  The same would be true if using other control modes (constant flow, constant pressure), etc...  A pump can be replaced or reset.  In addition, setting the intended speed in the upstream code now also turns on the pump which is not something I want in my HA UI.  I don't know if that's an artifact of the BLE API or just something the upstream implementation chooses to do.  It makes sense to store a pump speed in the pump if you're going to only use built-in pump scheduling since if the pump is going to turn itself on/off, it has to know what speed to run at.

## On/Off control
The upstream code sends the pump speed with every on/off command (when in constant speed mode, but works similarly in other modes).  This seems inconsistent with storing the pump speed in the pump as it requires you to have properly synced into the ESP32 what the desired pump speed (from the pump) is just so you can send it back to the pump.  Again, I don't know if this is a limitation of the BLE API.  It would be very oddly strange to have BLE control when all settings are stored in the pump that can't just command the pump to turn on or off as that's one of the most basic controls you would want.  Said differently, the current implementation for on/off is inconsistent with storing the intended pump speed (or the setting for whatever control mode you'ure using) in the pump.

## What the code actually does for control (confirmed from code + the author's protocol docs)
On/off, mode, and setpoint are all bundled into one Class 10 write: OpSpec 0x90, Sub 0x5600, Obj 0x0601, payload `2F 01 00 00 07 00 [Flag] [ModeByte] [Suffix(4)]`, where Flag 0x00 = start, 0x01 = stop, and Suffix is the float setpoint — or, if none is supplied, a hard-coded default (that's the bug where turn-on ignores the configured speed). There is no standalone on/off at this object; every start/stop carries the mode and a setpoint suffix. This confirms the On/Off concern above and pins the cause: it's the format of the control object, not just a coding choice.

## Possible cleaner on/off via GENIbus Class 3 commands
Start/stop are *standard GENIbus commands* (Class 3 "Commands") — not something you'd only learn by sniffing the phone app, which is itself a hint the project is still in the app-emulation phase. The code already speaks Class 3: remote enable/disable are `03 C1 07` / `03 C1 06`. So the pump accepts Class 3 command IDs, and the GENIbus standard command set typically includes START/STOP-type commands too. A bare `03 C1 <id>` might give a clean on/off with no mode or setpoint attached — exactly the primitive that's missing today. This needs an experiment on the pump to find which command IDs it honors, but if it works it's a much better on/off than the bundled Class 10 control object. Worth asking the maintainer directly: "have you tried the standard GENIbus START/STOP commands rather than the Class 10 control object?"

## Decoupling setpoint changes from on/off (likely a small fix)
The reason setting the speed also turns the pump on is that the set-speed routine leads with the start-asserting control-object write (Flag = start), then writes the dedicated setpoint. But there is already a separate setpoint-only write — Class 10 OpSpec 0x84, Obj 86, Sub 13 (speed) — that carries no run-state. Routing "adjust speed" through that write alone, instead of the control object, would likely let you change the setpoint without flipping on/off. Unverified: whether a standalone setpoint write while the pump is stopped actually takes effect. If it does, this is a small, clean fix and a good contribution candidate.

## Non-speed setpoint / unit issues
The constant-flow and constant-pressure setpoints come back garbage or with unit-conversion errors (part of broader telemetry scaling problems). RPM (constant speed) was chosen because it's effectively unitless — nothing to convert, nothing to get wrong. The decoder's scaling for the non-speed modes needs work; that's a concrete fix/contribution area independent of the architecture debate.
